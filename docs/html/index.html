What is Ralph?

"I'm helping!" — Ralph is an AI-agent-powered continuous build loop that turns your beads ticket queue into working, tested, committed code — one iteration at a time.

You write the tickets. You define the rules. Ralph builds the code.

🔄

Automatic Loop

Pulls the next ready ticket, invokes your AI agent, validates output, commits. Repeats 24/7.

🛡️

Guardrails Built-In

Preflight checks skip blocked/wrong-time tickets. Validation gate catches broken code before commit.

📊

Full Observability

JSONL metrics, health checks, daily/weekly reports. Know exactly what Ralph built and when.

🔌

Language Agnostic

Python, Node, Go, Rust, or custom. One ralph init scaffolds the right config for your stack.

The Problem Ralph Solves

Building software with AI agents normally requires you to be the orchestrator — picking tickets, pasting context, waiting, testing, committing, repeating. Ralph automates the mechanical steps so you focus on architecture, design, and review.

How It Works

Ralph runs a continuous loop. Each iteration has 6 steps:

  1. 1 PullQueries bd ready --json for unblocked open tickets. Skips epics/features (containers). Sorts deterministically: feature ascending, then task ascending.
  2. 2 PreflightIterates ready tickets through config/ralph_preflight.sh. First ticket that outputs READY gets selected. Others are skipped with a reason.
  3. 3 Assemble PromptBuilds an adaptive prompt: base (PROMPT.md) + type-specific guidance (prompts/bugfix.md) + build phase reference doc + task context.
  4. 4 Invoke AgentCalls kimi --print -p "..." or pi --print "...". The agent implements the task. Checkpoint saved before invocation for crash recovery.
  5. 5 ValidateRuns pytest (tiered: targeted by default), black, isort, flake8, mypy — only on changed files. All must pass.
  6. 6 Commit & RepeatIf worktree is clean, clears checkpoint. Logs metrics. Sleeps 5 seconds. Loops back to step 1.

What Ralph Does NOT Do

❌Write your tickets — you define them in beads
❌Design your architecture — you define that in AGENTS.md
❌Replace you as the developer — it replaces the mechanical steps
❌Run in production — it's a development tool

Beads Quick Reference

# Create tickets
bd new "P1: Implement data model" --type task --labels "phase-1"
bd new "[EXIT] P1: Integration tests" --type task --labels "exit,phase-1"

# Dependencies
bd dep add <child-id> <parent-id>
bd dep tree <id>

# Queue
bd ready                    # Unblocked tickets
bd ready --label phase-1    # Filter by label
bd ready --json             # Machine-readable

# Status
bd update <id> --claim
bd update <id> --status in_progress
bd update <id> --status closed --notes="All tests pass"
bd update <id> --status blocked --notes="Waiting for API key"

# Remember context across sessions
bd remember checkpoint "Last: phase-1 EXIT done"
bd recall checkpoint

Common Ticket Patterns

Sequential Build Phase

phase-1, meta-grouping (FEATURE)     ← Container
├── phase-1 (TASK) mybot.1.1         ← First task
├── phase-1 (TASK) mybot.1.2         ← Depends on 1.1
├── phase-1 (TASK) mybot.1.3         ← Depends on 1.2
└── exit, phase-1 (TASK) mybot.1.4   ← EXIT: depends on 1.1,1.2,1.3
                                       Blocks mybot.2.1

Parallel Work Within a Phase

phase-1, meta-grouping (FEATURE)
├── phase-1 (TASK) mybot.1.1         ← No deps (parallel with 1.2)
├── phase-1 (TASK) mybot.1.2         ← No deps (parallel with 1.1)
├── phase-1 (TASK) mybot.1.3         ← Depends on 1.1 AND 1.2 (merge)
└── exit, phase-1 (TASK) mybot.1.4   ← EXIT: depends on 1.3

ps -p $(cat .ralph_loop.pid)

Scenario 3: Agent Keeps Failing Validation

# Check what's failing
bash scripts/ralph/ralph_validate.sh --tier=targeted

black src/ tests/

# Block the ticket and move on

Cleanup & Restart

Full Cleanup

kill $(cat .ralph_loop.pid) 2>/dev/null || true
rm -f .ralph_loop.pid
rm -f .ralph_checkpoint.json
git status   # Should be clean

Reset and Restart

# 1. Kill everything
kill $(cat .ralph_loop.pid) 2>/dev/null || true
rm -f .ralph_loop.pid .ralph_checkpoint.json

# 2. Clean worktree
git checkout -- .
git clean -fd

# 3. Re-open in-progress tickets
bd list --status in_progress --json | python3 -c "
import json, sys, subprocess

FAQ

General

Q: Is this CI/CD?
No. Ralph is a development-time tool. It runs on your dev machine, not a CI server. Think of it as an autonomous pair programmer working through your backlog.

Q: Can I use Ralph without beads?
No. Ralph depends on bd for ticket management.

Q: Does Ralph work on Windows?
Use WSL2 (Windows Subsystem for Linux). Ralph is bash-based.

Q: How many tickets per day?
10-30 depending on complexity. Each iteration is 5-30 minutes.

Setup

Q: ralph command not found after install?

ls -la /usr/local/bin/ralph ~/.local/bin/ralph
source ~/.zshrc
echo $PATH | tr ':' '\n'

Q: Already have a project?

cd my-existing-project && ralph init
# Choose same directory — Ralph adds files alongside yours

Tickets

Q: Why are tickets skipped?
Check config/ralph_preflight.sh. Default skips epics and features (containers). Your rules may also block tickets based on labels or environment.

Q: How small should tickets be?
Completable in a single agent iteration (5-30 min). If a ticket cycles repeatedly, break it into two smaller tickets.

Troubleshooting

Q: Ralph keeps failing on the same ticket?

bash scripts/ralph/ralph_validate.sh --tier=targeted  # See what fails
bd update <id> --status blocked --notes="Manual fix needed"
rm -f .ralph_checkpoint.json
bash scripts/ralph/run_ralph_loop.sh

Q: Computer slept — what happens?
Agent pauses. On wake, continues. If agent times out, checkpoint is retained and loop resumes on next iteration.
