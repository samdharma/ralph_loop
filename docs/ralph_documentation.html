What is Ralph?

"I'm helping!" — Ralph is an AI-agent-powered continuous build loop that turns your beads ticket queue into working, tested, committed code — one iteration at a time.

You write the tickets. You define the rules. Ralph builds the code.

🔄

Automatic Loop

Pulls the next ready ticket, invokes your AI agent, validates output, commits. Repeats 24/7.

🛡

Guardrails Built-In

Preflight checks skip blocked/wrong-time tickets. Validation gate catches broken code before commit.

📊

Full Observability

JSONL metrics, health checks, daily/weekly reports. Know exactly what Ralph built and when.

🔌

Language Agnostic

Python, Node, Go, Rust, or custom. One ralph init scaffolds the right config for your stack.

How It Works

Ralph runs a continuous loop. Each iteration has 6 steps:

Tests + lint"] E --> F["6. COMMIT
and Repeat"] style A fill:#1f2937,stroke:#388bfd,color:#c9d1d9 style B fill:#1f2937,stroke:#d29922,color:#c9d1d9 style C fill:#1f2937,stroke:#a371f7,color:#c9d1d9 style D fill:#1f2937,stroke:#3fb950,color:#c9d1d9 style E fill:#1f2937,stroke:#f85149,color:#c9d1d9 style F fill:#1f2937,stroke:#58a6ff,color:#c9d1d9
  1. 1PullQueries bd ready --json for unblocked open tickets. Skips epics/features (containers). Sorts deterministically: feature ascending, then task ascending.
  2. 2PreflightIterates candidates through config/ralph_preflight.sh. First ticket that outputs READY gets selected. Others are skipped with a reason.
  3. 3Assemble PromptBuilds an adaptive prompt: base (PROMPT.md) + type-specific guidance + build phase reference doc + task context.
  4. 4Invoke AgentCalls kimi --print -p "..." or pi --print "...". The agent implements the task. Checkpoint saved before invocation for crash recovery.
  5. 5ValidateRuns pytest (tiered: targeted by default), black, isort, flake8, mypy — only on changed files.
  6. 6Commit & RepeatIf worktree is clean, clears checkpoint. Logs metrics. Sleeps 5 seconds. Loops back to step 1.

Single Iteration — Detailed Sequence

sequenceDiagram participant Loop as ralph_loop.sh participant Beads as beads (bd) participant PF as ralph_preflight.sh participant Agent as AI Agent participant Val as ralph_validate.sh participant Git as git Loop->>Beads: bd ready --json Beads-->>Loop: [ticket1, ticket2, ...] Loop->>Loop: Deterministic sort loop For each candidate Loop->>PF: Check labels + type PF-->>Loop: READY or BLOCKED alt READY Loop->>Loop: Select this ticket end end Loop->>Beads: bd update --claim + in_progress Loop->>Loop: Write checkpoint Loop->>Loop: Assemble adaptive prompt Loop->>Agent: Invoke with prompt Agent-->>Loop: Done Loop->>Val: bash ralph_validate.sh --tier=targeted alt Pass and clean Loop->>Loop: Clear checkpoint else Pass and dirty or Fail Loop->>Loop: Retain checkpoint end Loop->>Loop: Log metrics, sleep 5s

What Ralph Does NOT Do

❌Write your tickets — you define them in beads
❌Design your architecture — you define that in AGENTS.md
❌Replace you as the developer — it replaces the mechanical steps
❌Run in production — it is a development tool

Common Ticket Patterns

Sequential Build Phase

phase-1, meta-grouping (FEATURE)
  phase-1 (TASK) mybot.1.1  -- First task
  phase-1 (TASK) mybot.1.2  -- Depends on 1.1
  phase-1 (TASK) mybot.1.3  -- Depends on 1.2
  exit, phase-1 (TASK) mybot.1.4  -- EXIT: depends on 1.1,1.2,1.3
                                    Blocks mybot.2.1

Parallel Work Within a Phase

phase-1, meta-grouping (FEATURE)
  phase-1 (TASK) mybot.1.1  -- No deps (parallel with 1.2)
  phase-1 (TASK) mybot.1.2  -- No deps (parallel with 1.1)
  phase-1 (TASK) mybot.1.3  -- Depends on 1.1 AND 1.2 (merge)
  exit, phase-1 (TASK) mybot.1.4  -- EXIT: depends on 1.3

Stuck Session Recovery and Git Cleanup

When Ralph gets stuck, follow these procedures in order. Always verify git status between steps.

Quick Recovery — Loop Is Hung

# Step 1: Check what is running
cat .ralph_loop.pid 2>/dev/null && echo "PID exists" || echo "No PID file"
ps aux | grep ralph_loop

# Step 2: Kill the loop
kill $(cat .ralph_loop.pid 2>/dev/null) 2>/dev/null || true
sleep 2

# Step 3: Force kill if still alive
kill -9 $(cat .ralph_loop.pid 2>/dev/null) 2>/dev/null || true

# Step 4: Clean up runtime files
rm -f .ralph_loop.pid
rm -f .ralph_checkpoint.json

# Step 5: Verify git is clean
git status
# Should show: nothing to commit, working tree clean

Full Reset — Agent Left Dirty Worktree

WARNING: This discards ALL uncommitted changes. Use only when you are sure.
# Step 1: Kill everything
kill -9 $(cat .ralph_loop.pid 2>/dev/null) 2>/dev/null || true
rm -f .ralph_loop.pid .ralph_checkpoint.json

# Step 2: See what is dirty
git status
git diff --stat

# Step 3: If you want to KEEP any changes, stash them
git stash save "ralph-recovery-$(date +%Y%m%d-%H%M%S)"

# Step 4: Hard reset to last clean commit
git checkout -- .
git clean -fd

# Step 5: Verify clean
git status
# Should show: nothing to commit, working tree clean

# Step 6: Re-open in-progress tickets
bd list --status in_progress --json | python3 -c "
import json, sys, subprocess
try:
    tickets = json.load(sys.stdin)
    for t in tickets if isinstance(tickets, list) else []:
        tid = t.get('id','')
        if tid:
            subprocess.run(['bd','update',tid,'--status','open'])
except: pass
"

# Step 7: Restart
bash scripts/ralph/run_ralph_loop.sh

Recovering Lost Work from Checkpoint

If Ralph rolled back work and you want to recover it:

# See what commits exist (including rolled-back ones)
git reflog

# Find the commit hash of lost work
git reflog | head -20

# Recover a specific commit
git cherry-pick <commit-hash>

# Or create a branch from the lost commit to inspect
git branch recovery-branch <commit-hash>
git checkout recovery-branch
# Inspect the code, then
git checkout main
git cherry-pick <commit-hash>  # if you want to keep it

Cleaning Up After Multiple Failed Iterations

# Check how many commits Ralph made today
git log --oneline --since="today" --author="$(git config user.name)"

# Squash many small Ralph commits into one
# (Use with caution — rewrites history)
git rebase -i HEAD~10
# In the editor: keep first as 'pick', change rest to 'squash'

# Or just reset to a known good point and recommit
git reset --soft HEAD~5    # Undo last 5 commits, keep changes staged
git commit -m "Batch: Phase 1 work from Ralph loop"

# Clean up stale branches
git branch --merged | grep -v "main\|master" | xargs git branch -d 2>/dev/null || true

# Clean up remote tracking
git remote prune origin

General Git Workflow Scenarios

Scenario: You Want to Commit Ralph's Work and Push

# 1. Stop the loop temporarily
cat .ralph_loop.pid | xargs kill

# 2. Review what Ralph did
git status
git diff --stat
git log --oneline -5

# 3. If everything looks good, push
git push

# 4. Sync beads state
bd dolt push

# 5. Restart the loop
bash scripts/ralph/run_ralph_loop.sh

Scenario: You Need to Pull Remote Changes While Ralph Is Running

# 1. Stop the loop
cat .ralph_loop.pid | xargs kill
rm -f .ralph_loop.pid

# 2. Stash any uncommitted Ralph work
git stash

# 3. Pull remote changes
git pull --rebase
bd dolt pull

# 4. Restore stashed work
git stash pop

# 5. Restart
bash scripts/ralph/run_ralph_loop.sh

Scenario: Ralph Created a Commit You Do Not Want

# Undo the last commit, keep changes in working directory
git reset --soft HEAD~1

# Or remove the last commit entirely
git reset --hard HEAD~1

# If already pushed (dangerous — only if you are the only user)
git reset --hard HEAD~1
git push --force-with-lease

Scenario: You Made Manual Changes and Need to Commit Them

# While Ralph is stopped:
git add -A
git commit -m "Manual fix: description of what you changed"

# Important: update ticket status in beads
bd update <ticket-id> --status closed --notes="Fixed manually. Commit: $(git rev-parse HEAD)"

# Verify
git status   # Should be clean

# Resume
bash scripts/ralph/run_ralph_loop.sh

Scenario: Merge Conflict from Ralph and Manual Work

# 1. Stop Ralph
cat .ralph_loop.pid | xargs kill
rm -f .ralph_loop.pid

# 2. See conflicted files
git status

# 3. Resolve conflicts in each file
# Edit files, remove <<<<<<<, =======, >>>>>>> markers

# 4. Mark as resolved
git add <resolved-files>

# 5. Complete the merge
git commit -m "Merge: resolved conflict between Ralph and manual work"

# 6. Verify
git status   # Should be clean
bash scripts/ralph/ralph_validate.sh --tier=targeted

# 7. Restart
bash scripts/ralph/run_ralph_loop.sh

Scenario: Complete Disaster Recovery

Last resort. This throws away everything and clones fresh.
# 1. Kill everything
kill -9 $(cat .ralph_loop.pid 2>/dev/null) 2>/dev/null || true

# 2. Save any manual work you did
cd ..
cp -r my-project my-project-backup-$(date +%Y%m%d)

# 3. Clone fresh
rm -rf my-project
git clone <your-repo-url> my-project
cd my-project

# 4. Re-init Ralph (does not overwrite existing files)
ralph init

# 5. Restore any manual files from backup
cp my-project-backup-*/docs/agent/PROMPT.md my-project/docs/agent/
cp my-project-backup-*/config/ralph_preflight.sh my-project/config/

# 6. Re-init beads
bd init
bd dolt pull

# 7. Start fresh
bash scripts/ralph/run_ralph_loop.sh

FAQ

General

Q: Is this CI/CD?
No. Ralph is a development-time tool. It runs on your dev machine, not a CI server.

Q: Can I use Ralph without beads?
No. Ralph depends on bd for ticket management.

Q: Does Ralph work on Windows?
Use WSL2. Ralph is bash-based.

Q: How many tickets per day?
10-30 depending on complexity. Each iteration is 5-30 minutes.

Setup

Q: ralph command not found after install?

ls -la /usr/local/bin/ralph ~/.local/bin/ralph
source ~/.zshrc
echo $PATH | tr ':' '\n'

Q: Already have a project?

cd my-existing-project && ralph init

Tickets

Q: Why are tickets skipped?
Check config/ralph_preflight.sh. Default skips epics and features. Your rules may also block.

Q: How small should tickets be?
Completable in a single agent iteration (5-30 min). If a ticket cycles, break it into two.

Troubleshooting

Q: Ralph keeps failing on the same ticket?

bash scripts/ralph/ralph_validate.sh --tier=targeted
bd update <id> --status blocked --notes="Manual fix needed"
rm -f .ralph_checkpoint.json
bash scripts/ralph/run_ralph_loop.sh

Q: Computer slept — what happens?
Agent pauses. On wake, continues. If agent times out, checkpoint is retained.
