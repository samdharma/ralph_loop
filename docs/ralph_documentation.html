What is Ralph?

"I'm helping!" — Ralph is an AI-agent-powered continuous build loop that turns your beads ticket queue into working, tested, committed code — one iteration at a time.

You write the tickets. You define the rules. Ralph builds the code.

Architecture: Ralph is a global CLI tool installed at ~/.ralph/. Core build scripts live there. Your project carries only config files — no build system scripts in your repo. Clean Git history, simple onboarding.

🔄

Automatic Loop

Pulls the next ready ticket, invokes your AI agent, validates output, commits. Repeats 24/7.

🛡

Guardrails Built-In

Preflight checks skip blocked/wrong-time tickets. Validation gate catches broken code before commit.

📊

Full Observability

JSONL metrics, health checks, daily/weekly reports. Know exactly what Ralph built and when.

🔌

Self-Contained Repos

Build scripts in scripts/ralph/, config in config/. Clone, init beads, and build. Works with any language.

How It Works

Ralph runs a continuous loop. Each iteration has 6 steps:

Assemble context"] C --> D["4. AGENT
kimi or pi"] D --> E["5. VALIDATE
Tests + lint"] E --> F["6. COMMIT
and Repeat"] style A fill:#1f2937,stroke:#388bfd,color:#c9d1d9 style B fill:#1f2937,stroke:#d29922,color:#c9d1d9 style C fill:#1f2937,stroke:#a371f7,color:#c9d1d9 style D fill:#1f2937,stroke:#3fb950,color:#c9d1d9 style E fill:#1f2937,stroke:#f85149,color:#c9d1d9 style F fill:#1f2937,stroke:#58a6ff,color:#c9d1d9
  1. 1PullQueries bd ready --json for unblocked open tickets. Skips epics/features (containers). Sorts deterministically: feature ascending, then task ascending.
  2. 2PreflightIterates candidates through config/ralph_preflight.sh. First ticket that outputs READY gets selected. Others are skipped with a reason.
  3. 3Assemble PromptBuilds an adaptive prompt: base (PROMPT.md) + type-specific guidance + build phase reference doc + task context.
  4. 4Invoke AgentCalls kimi --print -p "..." or pi --print "...". The agent implements the task. Checkpoint saved before invocation for crash recovery.
  5. 5ValidateRuns pytest (tiered: targeted by default), black, isort, flake8, mypy — only on changed files.
  6. 6Commit & RepeatIf worktree is clean, clears checkpoint. Logs metrics. Sleeps 5 seconds. Loops back to step 1.

What Ralph Does NOT Do

❌Write your tickets — you define them in beads
❌Design your architecture — you define that in AGENTS.md
❌Replace you as the developer — it replaces the mechanical steps
❌Run in production — it is a development tool

Stuck Session Recovery

Quick Recovery — Loop Is Hung

cat .ralph_loop.pid 2>/dev/null && echo "PID exists" || echo "No PID file"
ps aux | grep ralph_loop
kill $(cat .ralph_loop.pid 2>/dev/null) 2>/dev/null || true
sleep 2
kill -9 $(cat .ralph_loop.pid 2>/dev/null) 2>/dev/null || true
rm -f .ralph_loop.pid .ralph_checkpoint.json
git status

Full Reset — Agent Left Dirty Worktree

WARNING: This discards ALL uncommitted changes.
kill -9 $(cat .ralph_loop.pid 2>/dev/null) 2>/dev/null || true
rm -f .ralph_loop.pid .ralph_checkpoint.json
git stash save "ralph-recovery-$(date +%Y%m%d-%H%M%S)"
git checkout -- .
git clean -fd
git status
# Re-open in-progress tickets
bd list --status in_progress --json | python3 -c "
import json, sys, subprocess
try:
    tickets = json.load(sys.stdin)
    for t in tickets if isinstance(tickets, list) else []:
        tid = t.get('id','')
        if tid: subprocess.run(['bd','update',tid,'--status','open'])
except: pass
"
ralph daemon

FAQ

How to use this FAQ: Questions are grouped by topic. Click any section to jump there from the sidebar.

General

Q: What is Ralph, in one sentence?
An AI-agent-powered continuous build loop that turns your beads ticket queue into working, tested, committed code — one iteration at a time.

Q: Is this CI/CD?
No. Ralph is a development-time tool. It runs on your dev machine, not a CI server. Think of it as an autonomous pair programmer working through your backlog.

Q: Can I use Ralph without beads?
No. Ralph depends on bd for ticket management. No tickets = nothing to build. The loop reads bd ready, bd show, and bd update to function.

Q: Does Ralph work on Windows?
Use WSL2 (Windows Subsystem for Linux). Ralph is bash-based and assumes a Unix-like environment.

Q: How many tickets can Ralph process per day?
10–30 depending on ticket complexity. Each iteration (one ticket) takes 5–30 minutes. A typical day: 15–20 iterations with 8–12 unique tickets.

Q: Where do Ralph scripts live? In my project or globally?
Both. The ralph CLI lives at ~/.ralph/bin/ralph (global). The 12 build scripts are copied into scripts/ralph/ inside each project during ralph init. This keeps each project self-contained and version-controlled.

Q: Can I run Ralph on multiple projects at once?
Yes. Each project has its own PID file (.ralph_loop.pid) and checkpoint. Start the daemon in each project directory and they run independently.

Setup & Initialization

Q: After ralph init, what do I need to do before starting the loop?
ralph init scaffolds infrastructure but does not create application code or tickets. Here is the complete post-init checklist:

  1. Set up dev environment — create .venv, install dependencies, create pyproject.toml
  2. Customize AGENTS.md — add your project's build commands, conventions, and design rules
  3. Customize docs/agent/PROMPT.md — update the Session Resumption Protocol with your reference docs, add project-specific design rules
  4. Create application spec files — docs/reference/ARCHITECTURE.md, BUILD_PHASE_N.md for each phase
  5. Create .env (if your project uses secrets or config)
  6. Set up git remote — git remote add origin <url>
  7. Create tickets in beads — epics, features, work tasks, exit tasks
  8. (Optional) Customize guardrails in config/ralph_preflight.sh
  9. (Optional) Set up config/TEST_MAP.yaml for targeted testing

Q: What files does Ralph check for at startup?
Three checks will exit the loop with an error if they fail:

CheckError Message
docs/agent/PROMPT.md existsERROR: PROMPT.md not found
AI agent (kimi or pi) in PATHERROR: No supported agent found
beads (bd) in PATHERROR: beads (bd) not found
Clean git worktreeERROR: Working tree has uncommitted changes

If config/ralph_preflight.sh is missing, Ralph uses its built-in defaults (skips epics/features) and does not fail.

Build Phase Docs

Q: What are BUILD_PHASE docs and why should I create them?
BUILD_PHASE docs (docs/reference/BUILD_PHASE_1.md, BUILD_PHASE_2.md, etc.) contain pre-researched API references for each build phase. When a ticket has a phase-N label, Ralph auto-injects the corresponding doc into the agent prompt. This saves 20–30 steps of redundant research per iteration.

Q: What should a BUILD_PHASE doc contain?

  • Goal: what this phase delivers
  • Dependencies: what must exist before starting this phase
  • API Reference: pre-discovered library types, method signatures, field names
  • Implementation patterns: code snippets showing the pattern to follow
  • Tests to write: specific test cases needed

Q: How does the auto-injection work?
Ralph extracts the phase number from the ticket's labels (e.g., phase-3 → 3), looks for docs/reference/BUILD_PHASE_3.md, and if it exists, injects it into the prompt with a note: "MANDATORY: Read this BEFORE researching APIs."

Q: Do I need BUILD_PHASE docs for every phase?
No, they are optional. Without them, the agent will discover APIs at runtime — it just takes more iterations. For phases where you've already researched the APIs, creating a BUILD_PHASE doc is a high-leverage 10-minute investment.

  • GATE_FAILED — agent produced code that doesn't pass tests or lint
  • BLOCKED — ticket skipped by preflight guardrails
  • ERROR — something broke in the loop itself
  • Same ticket ID appearing repeatedly — agent stuck in a loop on one task

Tickets

Q: Why are my tickets being skipped?
Check the preflight output in logs/ralph_loop.log. Common reasons: ticket has meta-grouping label (epic/feature containers are skipped by default), or a custom guardrail rule in config/ralph_preflight.sh is blocking it.

Q: How small should tickets be?
Completable in a single agent iteration (5–30 minutes). If a ticket cycles repeatedly (same ticket appears in consecutive iterations with GATE_FAILED), it is too large — break it into two smaller tickets.

Q: What is the ticket naming convention?
<PROJECT-SLUG>.<FEATURE-NUMBER>.<TASK-NUMBER>. Example: mybot.1.2 = Feature 1, Task 2. Feature and task numbers must be integers. The loop sorts deterministically by these numbers.

Q: Can I change a ticket while Ralph is running?
Yes, but stop the loop first. Use bd update <id> --status blocked to take it out of the ready queue, make your changes, then set it back to open. Restart the loop.

Troubleshooting

Q: Ralph keeps failing on the same ticket — what do I do?

# 1. See what's failing
ralph validate --tier=targeted

# 2. Fix manually if it's a simple lint/format issue
black src/ tests/
isort src/ tests/

# 3. If the task is too complex, block it and move on
bd update <id> --status blocked --notes="Needs manual intervention or splitting"
rm -f .ralph_checkpoint.json
ralph daemon

Q: My computer went to sleep while Ralph was running — what happens?
The agent process pauses. When the computer wakes, the agent continues. If the agent times out, the checkpoint is retained and the loop resumes from that checkpoint on the next iteration. No data loss — the checkpoint system handles this gracefully.

Q: How do I recover lost work from a checkpoint rollback?

# Ralph rolled back work — see what commits exist
git reflog

# Find the commit hash of lost work
git reflog | head -20

# Recover a specific commit
git cherry-pick <commit-hash>

# Or inspect it on a branch first
git branch recovery-branch <commit-hash>
git checkout recovery-branch
# If it looks good:
git checkout main
git cherry-pick <commit-hash>

Q: How do I collect debug data when reporting an issue?

echo "=== RALPH DEBUG ===" > /tmp/ralph_debug.txt
ralph version >> /tmp/ralph_debug.txt 2>&1
git status >> /tmp/ralph_debug.txt 2>&1
git log --oneline -10 >> /tmp/ralph_debug.txt 2>&1
git reflog -10 >> /tmp/ralph_debug.txt 2>&1
bd list >> /tmp/ralph_debug.txt 2>&1
tail -50 logs/ralph_loop.log >> /tmp/ralph_debug.txt 2>&1
ralph health --verbose >> /tmp/ralph_debug.txt 2>&1
cat .ralph_checkpoint.json 2>/dev/null >> /tmp/ralph_debug.txt || echo "(none)" >> /tmp/ralph_debug.txt
echo "Bundle saved to /tmp/ralph_debug.txt"

Advanced

Q: Can I customize which lint tools run in the validation gate?
Yes. Set RALPH_LINT_TOOLS to a space-separated list. Example: export RALPH_LINT_TOOLS="ruff mypy" to use ruff instead of flake8. Set it in your .env or shell profile.

Q: Can I use a custom agent prompt?
Yes. Either edit docs/agent/PROMPT.md directly, or point RALPH_PROMPT_BASE to a different file: export RALPH_PROMPT_BASE="docs/agent/my_custom_prompt.md".

Q: What happens if I add new tickets while Ralph is running?
Ralph queries bd ready at the start of each iteration, so new ready tickets are picked up on the next iteration automatically. No restart needed.

Q: How do I use Ralph in a team?
Ralph is designed for solo use with a shared git remote. One team member runs Ralph locally. Others pull the commits. Multiple people running Ralph on the same project will conflict — use branch isolation if needed.

Q: Does Ralph work with languages other than Python?
The ralph init wizard supports Python, Node.js/TypeScript, Go, Rust, and custom. The validation gate is currently Python-specific (ralph_validate.sh uses pytest/black/isort/flake8/mypy). For other languages, the gate generates as a stub that you customize. Multi-language validation scripts are planned for a future release.
